# Deploying the hosted instance — Render + Neon

How `minerva.thefarshad.com` + `lit.thefarshad.com` run now that `minerva-prod`
(the DigitalOcean droplet) is gone: the same `web/Dockerfile` image, on a free
Render web service, backed by a free Neon Postgres instead of a droplet-local
Postgres container. No VM to patch or pay for.

Not included in this deploy: the `minerva-services` helper (yt-dlp downloads +
PDF/arXiv import extras). The core planner, papers, sketches, sharing, and
Drive sync all work without it — add it later as a second Render service from
`docs/Dockerfile` if you want those extras back.

## 1. Neon Postgres

1. Create a project at [neon.tech](https://neon.tech) (free tier, no card,
   doesn't expire).
2. Dashboard → **Connection Details** → copy the **pooled** connection string
   (has `-pooler` in the hostname). The app already connects via `postgres.js`
   with `prepare: false` (see `web/src/db/index.ts`), which is required for
   Neon's PgBouncer pooler — no code change needed.
3. Restore the archived data (from `~/backups/thefarshad-sunset/minerva/db/`)
   before first boot, so the app's startup migration is a no-op:
   ```sh
   zcat minerva-postgres.sql.gz | psql "<pooled-connection-string>"
   ```
   Use the **unpooled** connection string for this one-off restore if the
   pooled one rejects the session-level commands `psql` sometimes issues.

## 2. Google OAuth — reuse the existing client

Minerva already has its own OAuth client (separate from Class++'s). In Google
Cloud Console → **APIs & Services → Credentials** → open that Web client →
**Authorized redirect URIs → Add**:
```
https://minerva.thefarshad.com/api/auth/callback/google
https://lit.thefarshad.com/api/auth/callback/google
```
(Add the Render-assigned `onrender.com` URL too if you want to test there
before the custom domains are live.) Keep the existing `GOOGLE_OAUTH_CLIENT_ID`
/ `GOOGLE_OAUTH_CLIENT_SECRET` — no new client needed.

## 3. Deploy on Render

Repo ships a Blueprint (`render.yaml`). In the Render dashboard: **New →
Blueprint → pick this repo**. Render builds `web/Dockerfile` and creates the
`minerva-web` web service, then prompts for the `sync: false` secrets:
`DATABASE_URL` (the Neon pooled string from step 1), `GOOGLE_OAUTH_CLIENT_ID`,
`GOOGLE_OAUTH_CLIENT_SECRET`. `AUTH_SECRET` is auto-generated by Render.

Health check: `https://<service>.onrender.com/api/version`.

Auth.js already runs with `trustHost: true` (see `web/src/auth.ts`) and infers
the callback origin from the request's `Host` header, so one service can
correctly serve sign-in on both `minerva.` and `lit.` — no per-domain
`NEXTAUTH_URL` juggling needed.

## 4. Custom domains

Both hostnames need to point at the same Render service:

1. Render service → **Settings → Custom Domains → Add** →
   `minerva.thefarshad.com`, then again for `lit.thefarshad.com`. Render shows
   a CNAME target (`<service>.onrender.com`) for each.
2. Cloudflare (`thefarshad.com` zone) → replace both hostnames' old A records
   (pointed at the now-destroyed droplet) with **CNAME**s to that target.
   Proxied is fine.
3. SSL/TLS mode **Full** (Render presents a valid cert).

Free plan **sleeps after ~15 min idle** (cold start ~20-30s on the next
request). Upgrade the plan if you want always-on.

## Everyday operations

- Logs: Render dashboard → the service → **Logs**.
- DB shell: `psql "<neon-connection-string>"`.
- Backup: `pg_dump "<neon-connection-string>" | gzip > minerva-$(date +%F).sql.gz`
- Rotate the session-signing secret (forces everyone to sign in again): Render
  dashboard → regenerate `AUTH_SECRET` → redeploy.
- Local dev / fully offline use is unaffected by any of this — see
  [`RUN-LOCAL.md`](RUN-LOCAL.md).
