Privacy Policy

Minerva — last updated 2026-05-08.

Minerva keeps your data in your own Google account and (optionally) on your own machine. The hosted SPA does not store, log, or transmit your data. An optional self-hosted helper service handles file downloads and a local mirror — its scope is documented below.

What Minerva is

Minerva is a static web app — HTML, CSS, and JavaScript files. The hosted instance at minerva.thefarshad.com is served from a small DigitalOcean droplet behind Caddy and Cloudflare. When you load it, the files run in your browser and call Google's APIs directly using your own OAuth grant. The droplet has no application database; it serves bytes and reverse-proxies a localhost-only helper service. You can also fork the repo and run the entire app yourself with no third-party hosting at all.

Operating modes

Settings → Connection lets you pick how data flows. Each mode has different storage:

  • Sheets-only (default): rows live in your Google Sheets spreadsheet and are mirrored to your browser's IndexedDB for offline reads. No other backend involved.
  • Hybrid: same as Sheets-only plus an optional mirror to a local Postgres provided by the helper. The mirror is a convenience cache; Sheets remains the source of truth.
  • Helper-assisted features (any mode): video downloads (yt-dlp), CORS-proxied bibliographic lookups (arxiv / CrossRef), local file save (~/Minerva on whatever machine runs the helper), and PDF text extraction. Each is opt-in by configuring a helper URL in Settings.

What is stored, and where

  • Your browser's localStorage: OAuth Client ID, Spreadsheet ID, access token (until it expires), email address (after first connect), helper URL, AI provider settings (only if you opt in), per-section preferences (sort, filter, group notes, group order). Cleared via Settings → Clear local config.
  • Your browser's IndexedDB: a mirror of your Minerva spreadsheet rows for offline reads, plus any video blobs you stored before switching to the host-disk path. Cleared via Settings → Clear local mirror, or Settings → Diagnostics → Reset offline copies.
  • Your Google Drive: a single spreadsheet named Minerva; an optional Minerva offline folder for downloaded videos when "Mirror downloads to Drive" is on; an optional minerva-config.json file (one JSON document) so a fresh device can sync your Settings on first sign-in.
  • The helper service's host disk (only if you've configured a helper URL): downloaded videos at ~/Minerva/videos/<playlist>/<title>.mp4; mirrored paper PDFs at ~/Minerva/papers/<title>.pdf; the helper's own logs (stdout from the container).
  • The helper's local Postgres (only with Hybrid mode + helper running): one row per spreadsheet row, keyed by (tab, id). Wipeable via Settings → Diagnostics → Reset offline copies (filesystem only — Postgres is wiped via the helper's own teardown).

That is the complete list. Nothing else is stored anywhere.

What requests your browser makes

When you use Minerva, your browser talks to:

  • Google (accounts.google.com, sheets.googleapis.com, www.googleapis.com, oauth2.googleapis.com) — for OAuth and to read/write your spreadsheet and (optional) Drive files. Subject to Google's privacy policy.
  • Google Fonts (fonts.googleapis.com, fonts.gstatic.com) — to load optional web fonts. Subject to Google's privacy policy.
  • jsDelivr (cdn.jsdelivr.net) — to load the QR-code library. Subject to the jsDelivr privacy policy.
  • Your configured helper URL (only if you set one in Settings) — for video downloads, CORS-proxied bibliographic lookups, file save, and PDF extraction. The helper is your software running on a host you control; this connection never reaches a Minerva-controlled server.
  • YouTube and arXiv embeds — when you preview a video or paper, your browser fetches their iframe content directly from those origins. Subject to their respective privacy policies.
  • Your configured CORS proxy (defaults to https://corsproxy.io/?; can be replaced with your helper's /proxy) — for arxiv and CrossRef metadata fetches that those servers do not return CORS headers for.
  • Cloudflare — the hosted instance is fronted by Cloudflare. Cloudflare sees request metadata (IP, URL, headers) for the static files it serves; this is the same as for any Cloudflare-fronted site. Subject to Cloudflare's privacy policy.

Minerva uses no analytics, no error reporters, no advertising trackers, no fingerprinting, and sets no cookies of its own. The hosted SPA runs no application logic on the server — the droplet's job is to serve static files and reverse-proxy the helper.

The optional helper service

The helper is a single Python script (docs/minerva-services.py) packaged as a Docker image. You decide whether to run it, where to run it, and what to point it at. Its responsibilities and the data it touches:

  • yt-dlp downloads (POST /download): you give it a URL, it spawns yt-dlp, streams the resulting media file back to your browser. The helper container can be configured to read your own browser cookies (--cookies-from-browser) for sites that require sign-in. Cookies are read at request time from a bind-mounted host directory; they are not copied or transmitted anywhere except to the target site (yt-dlp -> YouTube/etc.) for the request you triggered.
  • File save / serve / delete / reveal (/file/save, /file/serve, /file/delete, /file/reveal, /file/clean): writes/reads/removes files inside a single bind-mounted host directory (typically ~/Minerva). The helper refuses any path outside that directory.
  • CORS proxy (/proxy): forwards your browser's GET to a fixed allow-list of bibliographic hosts (export.arxiv.org, arxiv.org, api.crossref.org, www.youtube.com, youtube.com, youtu.be, api.semanticscholar.org). Other hosts are refused. The proxy does not log request bodies or store responses; it forwards bytes in both directions.
  • PDF extraction (/pdf/extract): runs opendataloader-pdf against a PDF URL or an uploaded PDF and returns the extracted markdown / structured JSON. Output goes to your browser; the helper deletes the temp file after the response.
  • Postgres mirror (/db/upsert, /db/delete, /db/rows, /db/dump): when "Mirror writes to Postgres" is on, the helper accepts JSON rows from your browser and stores them in a local Postgres container. The data is the same row data already in your spreadsheet — Minerva never sends anything else through this endpoint.
  • Health and shutdown (/health, /shutdown): probe and graceful stop. Both are loopback-only on a typical install.

The hosted instance's helper at minerva.thefarshad.com runs on the operator's DigitalOcean droplet and is reachable only via Caddy's reverse proxy. Anything stored on that helper's disk is on a machine the operator controls — see "Hosted instance, custody, and access" below.

Hosted instance, custody, and access

If you use the hosted instance at minerva.thefarshad.com instead of your own copy:

  • The droplet is operated by the maintainer (Farshad). The maintainer technically can read the droplet's filesystem, container logs, and helper-side caches when troubleshooting, the same way any sysadmin can read their own server. The maintainer does not, by policy, inspect this data, and there is no automated logging, mirroring, or backup of user data on the droplet.
  • If you upload videos / papers to the droplet's helper, those bytes briefly live on the droplet's disk under ~/Minerva/videos or ~/Minerva/papers. They are not encrypted at rest beyond what the droplet's filesystem provides. Treat the hosted helper the same way you would any other operator-controlled web service.
  • Cloudflare sits in front of the droplet and may cache static files. Cloudflare does not see the contents of authenticated Sheets/Drive API responses (those go from your browser straight to Google).
  • If you don't want to trust the hosted instance for file storage, leave the "yt-dlp server" / file-save fields blank in Settings and only use Sheets/Drive — or fork and self-host. The codebase is designed so the helper is strictly opt-in.

The public-share feature

When you build a card via Quick share, the title, body, and choices are encoded into the URL hash itself — they live in the URL, not on a server. Anyone with the URL sees the same card. Treat anything you put into a public share as public.

How Minerva interacts with Google user data

This section is the explicit, scope-by-scope disclosure required under the Google API Services User Data Policy and the Google APIs Terms of Service.

Scopes Minerva requests

  • https://www.googleapis.com/auth/drive.file
  • https://www.googleapis.com/auth/userinfo.email
  • openid

Minerva does not request any sensitive or restricted scopes. It does not request https://www.googleapis.com/auth/spreadsheets, full-Drive read, Gmail, Calendar, Contacts, or any other broader scope.

Data accessed via drive.file

What is accessed: a Google Sheets spreadsheet titled "Minerva", an optional folder titled "Minerva offline" containing video / paper files you explicitly chose to mirror, and an optional file titled "minerva-config.json". The drive.file scope is restricted by Google's design so an application can only see files it has itself created or that the user has explicitly opened with that application. Minerva is technically and contractually unable to access, list, search, read, or modify any other file in your Google Drive.

Within the Minerva spreadsheet, Minerva accesses sheet (tab) names and their contents (cell values you create — text, numbers, dates, links, etc.), sheet structural metadata (which tabs exist, their numeric sheet IDs), and the file's modifiedTime / webViewLink properties.

How it is used: render section list views, dashboards, and the calendar from your spreadsheet rows; write back to the spreadsheet when you edit a cell, add or delete a row, or install a section preset; on user-initiated actions, generate a public iCal file (Minerva tasks.ics) and an optional RSS file (Minerva tasks.rss.xml) inside the same Drive (these files are owned by the user, are explicitly created by clicking the in-app "Publish" button, and are made world-readable so the user can subscribe to them); on user-initiated actions, upload an annotated PDF or a downloaded video to the Minerva Drive folder; and (optionally) sync Settings via minerva-config.json.

How it is processed: all requests originate from your browser and go directly to Google's API servers using your own access token. The Minerva droplet does not see, intercept, or proxy these requests; it serves only static files plus the optional helper endpoints described above.

How it is stored: a copy of the spreadsheet's data is mirrored to your browser's IndexedDB for offline use. When the Postgres mirror is enabled and a helper is configured, that copy is also written to the helper's local Postgres. Both stores are wipeable from the in-app Settings.

How it is shared: Minerva does not share, sell, transfer, or disclose Google user data to any third party. The hosted droplet does not receive the data — Sheets/Drive traffic flows browser-to-Google.

Data accessed via userinfo.email and openid

What is accessed: your primary Google Account email address and the standard OpenID Connect identity claim that confirms you are authenticated.

How it is used: the email is displayed in the UI ("Connected as name@example.com"). The OpenID claim is used purely to confirm authentication; no inference, profiling, or analytics is performed on it.

How it is processed and stored: the email is fetched once after sign-in from https://www.googleapis.com/oauth2/v3/userinfo using your access token, then cached in localStorage. It is removed via Disconnect / Clear local config / clearing the site's data.

How it is shared: not shared, not transmitted to any third party, not stored anywhere outside your own browser.

Use of Google user data for AI / machine learning

Minerva does not use Google user data to develop, improve, or train any machine learning or generalized AI model. Google user data is also not transferred to any third party for those purposes.

Minerva includes an optional, opt-in AI assistant (Settings → AI assistant) that lets you supply your own API key for the AI provider of your choice. When you explicitly send a prompt, the prompt — which may include excerpts of your spreadsheet data — is sent directly from your browser to the endpoint you configured. Your API key is stored only in your browser. If the AI feature is not configured, no data ever leaves your browser for AI purposes.

Data retention and deletion

Minerva has no Minerva-controlled server-side storage; all retention and deletion is under your direct control:

  • Local mirror in IndexedDB: Settings → Local store → Clear local mirror, or clear the site's data in the browser.
  • Access token, email, OAuth client ID, helper URL: Settings → Disconnect or Clear local config, or clear the site's data.
  • Files on the helper's disk: Settings → Diagnostics → Reset offline copies (also accessible per-row via the kebab menu).
  • The Minerva spreadsheet / Minerva offline folder / minerva-config.json on Drive: you own them; delete them from Drive at any time. Minerva does not need to be involved.
  • Authorization grant: revoke at Google Account → Apps with access at any time.
