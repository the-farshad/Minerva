Privacy Policy

Minerva — last updated 2026-04-29.

Minerva does not collect, store, or transmit any of your data. There are no servers because there is no backend. Your data lives in your Google account, owned and accessed by you.

What Minerva is

Minerva is a static web app — HTML, CSS, and JavaScript files served from GitHub Pages. When you load it, the files run in your browser and call Google's APIs directly using your own OAuth grant. There is no Minerva server in between.

What is stored, and where

  • In your browser's localStorage: your OAuth Client ID, your spreadsheet ID, your access token (until it expires), your email address (after first connect), and your theme/font preferences. Cleared when you click Clear local config in Settings or clear this site's data in your browser.
  • In your browser's IndexedDB: a mirror of your Minerva spreadsheet, kept for offline reads and fast rendering. Cleared when you click Clear local mirror in Settings or clear this site's data.
  • In your Google Drive: one spreadsheet named Minerva, created by Minerva when you first connect. Owned by you. Never read or written by anyone but you.

That is the complete list. Nothing else is stored anywhere.

What requests your browser makes

When you use Minerva, your browser talks to:

  • Google (accounts.google.com, sheets.googleapis.com, www.googleapis.com, oauth2.googleapis.com) — for OAuth and to read/write your spreadsheet. Subject to Google's privacy policy.
  • Google Fonts (fonts.googleapis.com, fonts.gstatic.com) — to load the optional web fonts (Inter, Roboto, Ubuntu, Vazirmatn, Atkinson Hyperlegible, VT323). Subject to Google's privacy policy.
  • jsDelivr (cdn.jsdelivr.net) — to load the QR-code library. Subject to the jsDelivr privacy policy.

Minerva uses no other third parties. There are no analytics, no error reporters, no advertising trackers, no fingerprinting, no cookies set by Minerva itself. Minerva receives no data because Minerva runs no server.

The public-share feature

When you build a card via Quick share, the title, body, and choices are encoded into the URL itself — they live in the URL hash, not on a server. Anyone with the URL sees the same card. The URL is generated locally; nothing is uploaded. Treat anything you put into a public share as public.

How Minerva interacts with Google user data

This section is the explicit, scope-by-scope disclosure required under the Google API Services User Data Policy and the Google APIs Terms of Service. It documents every type of Google user data Minerva accesses, exactly how that data is used, where it is stored, and whether it is shared.

Scopes Minerva requests

  • https://www.googleapis.com/auth/drive.file
  • https://www.googleapis.com/auth/userinfo.email
  • openid

Minerva does not request any sensitive or restricted scopes. It does not request https://www.googleapis.com/auth/spreadsheets, full-Drive read, Gmail, Calendar, Contacts, or any other broader scope.

Data accessed via drive.file

What is accessed: A single Google Sheets spreadsheet titled "Minerva" inside your Google Drive. The drive.file scope is restricted by Google's design so that an application can only see files it has itself created or that the user has explicitly opened with that application. As a result, Minerva is technically and contractually unable to access, list, search, read, or modify any other file in your Google Drive — including documents, photos, sheets, slides, or any other content you did not create through Minerva.

Within the Minerva spreadsheet, Minerva accesses the following content types:

  • Sheet (tab) names and their contents (cell values you create — text, numbers, dates, links, etc.).
  • Sheet structural metadata (which tabs exist, their numeric sheet IDs) so Minerva can add or remove rows/tabs you create through the app.
  • The file's "modifiedTime" and "webViewLink" properties, used to display a link to your spreadsheet from the Minerva interface.

How it is used: The accessed data is the user's own task / goal / project / note / habit data, displayed and edited within the Minerva user interface. The app reads the spreadsheet to render section list views, dashboards, and the calendar; writes back to the spreadsheet when the user edits a cell, adds a row, deletes a row, or installs a section preset; and, on a user-initiated action, generates a public iCal file (Minerva tasks.ics) and an optional RSS file (Minerva tasks.rss.xml) inside the same Drive — these files are owned by the user, are explicitly created by the user clicking the in-app "Publish" button, and are made world-readable so the user can subscribe to them from a calendar or RSS reader.

How it is processed: All requests originate from the user's web browser and go directly to Google's API servers (sheets.googleapis.com, www.googleapis.com) using the user's own access token. The Minerva web app is a static site (HTML, CSS, JavaScript files served from GitHub Pages); it has no server-side component, no proxy, no backend database, and no logging or analytics infrastructure. The Minerva domain (minerva.thefarshad.com) does not receive any of the data Minerva reads from or writes to your Google Drive.

How it is stored: A copy of the spreadsheet's data is mirrored into the user's own browser using IndexedDB (under origin https://minerva.thefarshad.com), so the app can render data offline and respond instantly to edits. This local mirror is created and stored entirely on the user's device. It is never transmitted anywhere except, on user action, back to the user's own Google Drive via the Sheets API. The user can delete this local mirror at any time via the in-app *Settings → Local store → Clear local mirror* button or by clearing the site's data through their browser's settings.

How it is shared: Minerva does not share, sell, transfer, or disclose Google user data to any third party. Minerva itself does not receive the data (it has no server). The hosted instance at minerva.thefarshad.com serves only static files; the operator of thefarshad.com has no programmatic access to user data.

Data accessed via userinfo.email and openid

What is accessed: The user's primary Google Account email address (a single string, e.g. name@example.com) and the standard OpenID Connect identity claim that confirms the user is authenticated.

How it is used: The email address is displayed in the user interface only — for example "Connected as name@example.com" on the Settings page and "Welcome back, name@example.com" on the home page. This is so the user knows which Google account is currently signed in. The OpenID claim is used purely to confirm authentication; no inference, profiling, or analytics is performed on it.

How it is processed: The email is fetched once after sign-in from https://www.googleapis.com/oauth2/v3/userinfo using the user's access token. The request originates from the user's browser; no Minerva server is involved.

How it is stored: The email is cached in the user's browser localStorage alongside the access token, so the UI can display "Connected as <email>" without repeating the userinfo request on every page load. It is removed when the user clicks Disconnect or Clear local config, or when the user clears the site's data through their browser.

How it is shared: The email is not shared, transmitted to any third party, or stored anywhere outside the user's own browser.

Use of Google user data for AI / machine learning

Minerva does not use Google user data to develop, improve, or train any machine learning or generalized AI model. Google user data is also not transferred to any third party for those purposes.

Minerva does include an optional, opt-in AI assistant feature (Settings → AI assistant) that lets the user supply their own API key for an AI provider of their choice (Anthropic, OpenAI, a local Ollama instance, or a custom OpenAI-compatible endpoint). When the user explicitly asks the assistant a question, the user's prompt — which may include excerpts of their Minerva spreadsheet data — is sent directly from the user's browser to the configured AI provider. This transmission only happens at the user's explicit action (clicking *Send* or pressing ⌘/Ctrl+Enter in the assistant overlay) and only goes to the endpoint the user themselves configured. The user's API key is stored only in the user's browser and is never transmitted to minerva.thefarshad.com or any other Minerva-controlled location. If the AI feature is not configured, no data ever leaves the user's browser for AI purposes.

Data retention and deletion

Because Minerva has no server-side storage, all retention and deletion is under the user's direct control:

  • Local mirror in IndexedDB: cleared via Settings → Local store → Clear local mirror, or by clearing the site's data in the browser.
  • Access token, email, OAuth client ID: cleared via Settings → Disconnect or Clear local config, or by clearing the site's data.
  • The Minerva spreadsheet in Google Drive: the user owns it and can delete it from Drive directly at any time. Minerva does not need to be involved.
  • Authorization grant: the user can revoke Minerva's access at Google Account → Apps with access at any time.
